# データアクセス方針の検討（Study・AI対戦）

**方針の軸（議論が迷走しないようにする一文）**

> **学習（Study/AI）は“快適性・軽さ”を最優先し、競技（Ranked/Friend）は“公平性・整合性”を最優先する。**

この前提で、Study と AI対戦のデータの持ち方と、実装・運用コストを検討する。

---

## 現状の整理

| 用途 | データの取得元 | 負荷のイメージ |
|------|----------------|----------------|
| **ランクマ・友達対戦** | Firestore（マッチ・問題） | 対戦のたびにアクセスする前提で許容 |
| **Study 4択問題一覧** | **アプリ内** `data/corpus-questions.json`（実装済み。不足時のみ Firestore にフォールバック） | 通常は read 0 |
| **Study ディクテーション** | **アプリ内** `data/dictation-vocab.json` | 既にバンドル済み。Firestore は未使用 |
| **AI対戦（4択）** | **アプリ内** `data/corpus-questions.json`（実装済み。id は `local-{level}-{index}`） | 問題取得の read 0。不足時はサーバーで問題取得 |

---

## 結論（方針として強い）

- **Ranked / Friend** … Firestore（サーバー側が主）。公平性・整合性のため現状どおり。
- **Study / AI** … アプリ内（クライアント側が主）。快適性・軽さのため。

**“同じ問題プールに揃える”は今は必須にしない。** 必須にすると一源化・デプロイ手順が一気に重くなる。ユーザーが増え、競技性を強くしたくなったタイミングで検討すれば十分。

---

## 1) Study 4択一覧をアプリ内保持：賛成（最優先で効く）

- **現状の痛み**: `limit(500)` × レベル数 の読み取りは無駄が大きく、一覧表示が重い。Study 一覧は「競技の公平性」も「サーバー整合性」も不要なので、オフラインで完結してよい領域。
- **方針**: 4択問題を JSON 等でアプリにバンドルし、Study の一覧はそこだけを参照する。Firestore は不正解リスト用のマッチ履歴など、ユーザー固有データのみに使う。
- **効果**: 一覧表示の Firestore read が 0 になり、体感も軽くなる。**今の痛みを最短で消せる。**

---

## 2) AI対戦をアプリ内参照：賛成（表現の修正あり）

- **効果**: 問題本文をアプリ内から参照すれば、プレイ中の Firestore read が減り、**負荷軽減と UX（テンポ・安定性）の改善**になる。レートが動かない前提ならアプリ内問題で完結させてよい。
- **“安全”について**: 「安全が向上する」はここでは正確ではない。  
  - アプリ内に置く = Firestore read が減って**安定・コスト減**。  
  - 一方、アプリ内の問題はユーザーが解析しようと思えば見える。  
  - 「正解を隠す」ことが目的ならアプリ内は不利。  
  - AI対戦は対戦相手がいないので公平性不要であり、正解が見え得る前提で問題ない。  
  → **安全ではなく、負荷と UX の改善**と表現する。

---

## 3) 一源化（二重管理）問題：深刻度を上げて方針で切る

- **問題**: Ranked/Friend は Firestore の `questions`、Study/AI はアプリ内 JSON とすると、「同じ問題プールに揃える」を必須にした瞬間に、一源化・ビルド手順・デプロイが重くなる。
- **方針（現実的な結論）**  
  - **Ranked / Friend** … Firestore（サーバー側が主）。  
  - **Study / AI** … アプリ内（クライアント側が主）。  
  - **“同じ問題プールに揃える”は必須要件にしない。**  
- **理由**: 必須にすると運用が一気に重くなる。競技性を強くしたくなったタイミングで「同じプールに揃える」を検討すれば十分。今は「学習は快適・軽く、競技は公平・整合」で分けておく。

---

## 4) 実装と管理運用コストの観点

### 実装コスト

- **Study 4択一覧**  
  - 4択問題をアプリ内に持つデータ形式（JSON 等）の用意、Study 画面の参照先を Firestore → ローカルに切り替える。  
  - 既存の `questions` からエクスポートするスクリプトか、既存のビルド（例: `build-dictation-vocab.js` と同種）の追加。  
  - 工数は中程度。**効果（read 削減・体感軽さ）が大きいので最優先で効く。**

- **AI対戦**  
  - マッチ作成 CF で「問題 ID リスト」を返す代わりに、クライアント側で「レベル＋アプリ内インデックス」から問題を選ぶ形に変更。  
  - または CF は「レベル・問題数」だけ返し、クライアントでアプリ内データからランダムに選択。  
  - 問題データは Study 用と共通化すれば、追加データ管理は 1 本で済む。

### 管理運用コスト

- **問題の追加・更新**  
  - Study/AI 用はアプリ更新（ストア申請）が必要。  
  - Ranked/Friend 用は Firestore の `questions` を更新するだけ（アプリリリース不要）。  
  - **今は「同じプール」を必須にしないので、両者は別サイクルでよい。** 学習用は更新頻度を抑え、競技用だけ柔軟に差し替える運用が現実的。

- **一源化する場合（将来オプション）**  
  - 問題マスタを JSON 等 1 ソースで持ち、ビルド時に「Firestore 投入」と「アプリバンドル」の両方に流す。  
  - デプロイ手順（CF とアプリのどちらを先に出すか、不整合期間の扱い）を決める必要があり、運用コストが増える。  
  - **現時点では必須にせず、競技性を強める段階で検討。**

## まとめ

| 項目 | 方針 |
|------|------|
| **なぜ分けるか** | 学習（Study/AI）は快適性・軽さ、競技（Ranked/Friend）は公平性・整合性を最優先するため。 |
| **Study 4択一覧** | アプリ内保持に移行。最優先で効く。 |
| **AI対戦** | アプリ内参照で負荷・UX 改善。「安全」ではなく負荷と UX。 |
| **問題プール一致** | 今は必須にしない。運用が重くなりすぎる。必要になったタイミングで検討。 |

このメモを方針の基準として使い、実装時は「学習はアプリ内・競技は Firestore」で一貫させるとブレない。

## 運用・管理で必要なこと

### データの所在と更新の責任

| データ | ファイル／場所 | 更新方法 | 反映タイミング |
|--------|----------------|----------|----------------|
| **Study 4択・AI 4択** | `data/corpus-questions.json` | JSON を編集またはスクリプトで生成 | アプリをビルドし直し・ストア申請 |
| **Study ディクテーション** | `data/dictation-vocab.json` | `node scripts/build-dictation-vocab.js` で再生成 | 同上 |
| **Ranked/Friend 用問題** | Firestore `questions` コレクション | Console または Admin SDK でドキュメント追加・更新 | 即時（アプリ更新不要） |

- **学習用（Study/AI）**: アプリにバンドルされるため、**問題を変えたらアプリの再ビルド・再リリースが必要**。更新頻度は抑え、大きな見直しやレベル追加のタイミングでまとめて変えるとよい。
- **競技用（Ranked/Friend）**: Firestore だけ変えればよい。不具合問題の差し替えや新問追加は、アプリを出し直さずに対応できる。

### 学習用データを更新するときの手順

1. **4択（corpus-questions.json）**
   - 形式: `{ "1": [ { "lang", "exam", "level", "prompt", "choices", "answerIndex", "explanation", ... } ], "2": [ ... ], ... }`（レベル "1"～"10" がキー）。
   - 既存キーに要素を足す／既存要素を直す／レベルを増やす、いずれも JSON 編集で対応。
   - 編集後はアプリをビルドし直し、動作確認してからストアに提出。
2. **ディクテーション（dictation-vocab.json）**
   - `node scripts/build-dictation-vocab.js` を実行して再生成（必要なら `--with-definitions`）。
   - 生成後はアプリをビルドし直し、ストア申請。

### デプロイ時の注意

- **Cloud Functions**: `submitAnswer` で `questionId` が `local-` 始まりのときは Firestore を読まず `correctChoiceIndex` で正誤判定。`createMatch` で AI 時に `questionIds` が渡ればそのまま使用。CF を先にデプロイしてからアプリを出しても問題ない。
- **アプリ**: `corpus-questions.json` を変えたら必ずビルドし直す。古いバンドルのまま出すと「ローカルに問題がない」と判定され、Study 一覧や AI 4択が Firestore にフォールバックしたり、AI でサーバー側問題取得に落ちる。

### トラブル時の確認ポイント

- Study 一覧が空／重い → 該当 TOEIC レベルの `corpus-questions.json` のキー（"1"～"10" の範囲）に問題が入っているか確認。なければ Firestore フォールバックになる（ネットと read が発生）。
- AI 4択で「問題が見つかりません」→ そのマッチの `questionIds` が `local-*` かどうか。`local-*` ならクライアントの `getQuestionById` で取れるはずなので、レベル・件数不足で `getRandomQuestionIdsForToeic` が空になっていないか確認。
- 競技用だけ問題を差し替えたい → Firestore `questions` の該当ドキュメントを更新または追加。アプリの再リリースは不要。

### 参照するコード（変更時に触りがちな場所）

| 役割 | ファイル |
|------|----------|
| 学習用問題の読み方・id 解決 | `lib/study-questions.ts` |
| Study 一覧の取得（ローカル優先→フォールバック） | `app/(tabs)/study.tsx`（4択一覧の useEffect） |
| AI マッチ開始時に questionIds を渡す | `app/(tabs)/battle.tsx`（`startAIMatch`） |
| 対戦中の問題取得（local ならローカル） | `app/match/[id].tsx`（`loadFirstQuestion` と onSnapshot 内の 2 箇所） |
| AI で questionIds を受け取る／local 時は correctChoiceIndex で採点 | `functions/src/index.ts`（`createMatch`, `submitAnswer`） |

### 将来「同じ問題プールに揃える」をやる場合

- 問題マスタを 1 ソース（例: JSON または Firestore）にし、**ビルド／デプロイ時に**「Firestore に流し込む」と「アプリ用 JSON を吐く」の両方を行う手順を用意する必要がある。
- デプロイ順序（CF とアプリのどちらを先に出すか）と、不整合期間の扱い（旧アプリが新 Firestore を読む／新アプリが旧 Firestore を読む）を決めてから運用に入ること。
